[2021/9/8] 캐치 개발자 커리어콘

1. 적극적 구직, 소극적 구직
    1. 언제나 구직 상태여야 한다고 생각. 더 좋은 기회가 올 수 있으니!
    2. 많이 만나보면 내 range를 알 수 있다.
    3. 일할 때도, 구직할 때도 회사보다 내가 우선!
        1. 내가 소중하다는 생각을 하자. 을의 입장이라 생각하지말자.
2. 이직을 하기로 결정햇다면
    1. 업무 문서 작성에 힘을 쏟자
        1. 이미 익숙한 기술이라도 다시 공부하며 문서로 기록
        2. 업무 시간에서, 문서 작성 시간을 늘리자
            1. 기술 스택 정리, 트러블 슈팅 정리, 튜토리얼 작성
    2. 포트폴리오 초안 작성
        1. 대학교 ~ 직장인 시기의 모든 기록을 작성해보기
            1. 까먹고 있었던것도 찾을 수 있음. 과거를 돌아보는 시간을 갖자.
        2. 지금 나의 상황을 진단하고 이직 준비 플랜을 설정하기 위함
    3. 채용 공고 틈틈히 리서치
        1. 갈 수 있을 것 같은 회사는?
        2. 내가 가고 싶은 회사는?
        3. 목표와 상황의 gap 파악 => 목표 조정
    4. 이직 목표를 세웠으면
        1. 포트폴리오 작성
        2. 이력서 작성
        3. 몇 개월 후의 포트폴리오로 미리 만들 수 있음
            1. 앞으로 할것들도 정리하면 체계적인 이직준비가능
3. 이력서와 포트폴리오는 뭐가 다를까?
    1. 이력서는
        1. 경력 기술서에 가깝다.
        2. 실제 업무 위주로 요약정리한것
        3. Fact만 작성
        4. 깔금하게 1-2장으로 정리하는 것이 Good
    2. 포트폴리오는 자기소개서의 확장
        1. 팩트 기반이지만
        2. 포장을 할 수 있고, 스토리텔링이 가능
        3. 바꾸기 어려운 이력의 부족함을 충분히 메울 수 있는 도구
    3. 따로 보내주는 것이 더 낫다
        1. 이력서는 pdf로 깔끔하게 제출
        2. 포트폴리오는
            1. Pdf -> 이력서와 같이 첨부
            2. 노션 -> 이력서에 링크 첨부
4. 게으른 사람의 사이드 프로젝트
    1. 90%의 실패와 10%의 성공
        1. 스스로 세운 계획은 거의 다 실패했지만
        2. 강의, 비강제적 스터디, 책완독 등등
        3. 딱 한번만 성공하면 쉬워진다.
            1. 나에게 맞는 방법 찾기
    2. 효과 있던 케이스
        1. 지인이 “이런거 만들어 줄수있어?”라고 물어보면, “응 내가만들어줄께”라고 대답하기
        2. 팀메이트 구해서 눈치보며 개발하기
        3. 유료샤이드프로젝트모임 신청하기
        4. 멘토링 받기
    3. 개발보다 기록을 더 노력
        1. 과거 / 현재 / 미래의 내가 협업한다고 생각. 미래의 나는 ‘일주일 후의 나’일수도… 모든 것 까먹은 ‘한달후’의 나일수도 있다.
        2. 기록에 집중!
        3. 기록은 일정 관리 x, checklist 0
        4. ‘미래의 내’가 읽고 10초만에 작업할 수 있는 수준으로 작성
    4. 회사 기술 스택 그대로 사용한 프로젝트
        1. 시작하는데 드는 에너지가 적고
        2. 바로 배운 내용을 적용할 수 있어서
        3. 이직 면접 준비를 동시에
    5. 회사 프로젝트 branch 따서 기술 스택 추가
        1. 적용하고 싶은 기술 스택
        2. 회사의 동의를 구하기 어렵다면
            1. 스터디용 branch를 따서 적용
    6. 사이드 프로젝트의 목적은 성장 > 완성
        1. 완성을 목표로 하지 않기
        2. 10분이라도 했으면 스스로 칭찬하기
        3. 배움과 성장에 집중하기
        4. 흥미가 떨어져서 하기가 싫다면, 과감히 포기하고 다른 프로젝트 시작하기
        5. 쉬운 목표로 조정하기
5. Notion으로 포트폴리오 만들기
    1. notion은 베이스로 활용
        1. 처음에 만들기 쉬운편
        2. 업데이트와 관리가 쉽다.
        3. 내용을 많이 담을 수 있다.
        4. 디자인 신경 많이 안써도 된다. 기본적으로 깔끔.
    2. 그외의견
        1. 너무 숨겨져있으면, 하나하나 펼쳐봐야하고 가독성 안좋은것같다. 쭈루룩 볼수있게
        2. 자기소개가 먼저 들어가면 느낌이 좋다.
        1. Notion 주의할점
        1. 채용담당자가 모든 글을 읽지 못한다.
        2. 어필하면 좋을 내용은 반드시 위에!
        3. Depth를 너무 만들지 말자
        4. 가독성에 신경써주자 (위아래 간격, 중요 글자 강조)
1. Github 으로 포트포리오 만들기
    1. 프로필 부분을 일단 꾸며봅시다.
        1. 화려하면 일단 관심이 생긴다.
        2. 채용담당자를 내가 원하는 방향으로 유도
        3. 내가 보여주고 싶은 것만 보여주자.
        4. Github 관리에 대한 부담을 줄이자.
    2. 개인 프로젝트가 없다면
        1. 회사 소스코드 중 주요 로직들을 보안 이슈 없도록 수정해서 (변수 명을 변경하는 등) Code snippet으로 정리해두면 어떨까?
2. 더 좋은 포트폴리오 만드는 TIP
    1. 나를 소개하는 문장 넣기
        1. 시작부터 내가 실력있는 개발자로 보이도록 나를 설명하는 문장 넣어주기
    2. 포트폴리오에 더 에너지를 쏟아보자
        1. 가성비가 좋다
        2. 프로젝트 하나하나에 내용 많이 담기
            1. 일단 내용이 많으면
            2. 정성과 성의가 느껴짐 => 호감
            3. 일을 많이 했다고 느껴짐 => 실력 좋아 보임
            4. 일을 열심히 한 것처럼 보임 => 신뢰, 믿음직
        3. 트러블 슈팅 사례 적으면 Good
            1. 적을 내용이 많다
                1. 문제 > 원인 > 대안 > 해결 > 평가 > 배운점
            2. 스토리 텔링가능
        4. 줄글 NO! 요약해서 정리
            1. 주요내용에 굵은표시 해주면 좋다.
        5. 회사마다 기술 스택을 다르게 강조하자.
            1. 회사가 원하는 기술스택과 기능 개발경험이 있는지가 가장 중요하니까..!
            2. 지원 하기전 채용 공고 확인 => 내 포트폴리오 순서 수정
    3. 블로그 포스팅 개수가 중요한 건 아니다.
        1. 10개 쓸 시간에 2개에 팍팍 쓰자.
    4. 삽질,실패한 경험도 적으면 좋을까요?
        1. 모든것은 실패는 없다.
        2. 하던 일련의 과정들은 지식이다. 이 실패한 경험을 가져간다는것도 좋은거다.
        3. 실패를 했다고 깨달아서 (당당하게) 이 실패덕분에 이걸 알게됐다. 더 성장했다. 아니면 굳이 실패했다고 안말하던가. 실패가 중요한게아니여서 실패라는 워딩을 빼도 좋을듯 .
